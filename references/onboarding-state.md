# `.roadway/onboarding.md` — progress state file

A single Markdown file that records what Roadway onboarding work has already been done on a codebase. Written by the `roadway-integrator` router and its sub-skills; read by the router on resume so a second session picks up where the first left off without re-asking.

## Location and lifecycle

- **Path:** `.roadway/onboarding.md` at the root of the customer repo. Create the `.roadway/` directory if missing.
- **Git:** do not add a `.gitignore` entry. The customer decides whether to commit.
- **Creation:** the router creates the file on first run, after the Detection pass, with the `# Roadway onboarding` header, the intro paragraph below, and the `## Detected stack` block.
- **Deletion:** the plugin never deletes the file. The customer (or the router's "start over" branch) does.
- **Resetting a step:** deleting a `## <skill> — done …` section makes the plugin treat that skill as not-yet-run.

## File shape

```markdown
# Roadway onboarding

_Generated by the roadway-integrator plugin. Safe to edit by hand — the plugin treats this file as source of truth for "what's already done." To redo a step, delete its section._

## Detected stack
- CRM: HubSpot (portal 12345)
- Analytics: GA4 (G-ABC123), no Segment
- Auth/signup entry: src/server/auth/signup.ts:42

## install-roadway-tag — done 2026-04-23
```

## Sections

### `## Detected stack` (router-owned)

Written once, on first run, by the router after the Detection pass. Never rewritten. One bullet per row of the detection table (CRM, Analytics, Auth, Payments, HTML host). Include parenthetical identifiers the customer might need to re-reference (portal IDs, measurement IDs, file paths).

### `## <skill-name> — <status>` (one per sub-skill)

`<skill-name>` is the skill's directory name (e.g. `setup-hubspot-crm`). `<status>` is one of:

- `in progress` — skill has been started but has unchecked items remaining.
- `done YYYY-MM-DD` — every item is checked; absolute date of completion.

Skills that haven't run yet have no section.

Inside a skill's section, use two subheadings when both categories apply:

- **External checkpoints** — one `- [x]` or `- [ ]` per mutation outside the repo: a custom property created in a CRM portal, a GA4 admin toggle, a Salesforce field mapping, a GTM publish. Mark each one as soon as the user confirms it is done, not at the end of the skill.
- **Code edits** — a single rolled-up `- [x]` for the in-repo diff the skill produced (e.g. "Hidden fields wired into 3 forms"). One bullet is fine; `git diff` holds the detail. Mark this when the code edits are complete.

Skills with no external mutations (e.g. `setup-segment-identify`, `setup-stripe-metadata`) omit the **External checkpoints** subheading — just list `- [x]`/`- [ ]` items directly under the section header.

## Checkpoint rubric — external vs code edit

**External** = any mutation the plugin cannot see with `git diff` in the customer's repo. Examples:

- HubSpot / Salesforce property or field created in the portal.
- GA4 Reporting Identity, DebugView, or other Admin toggle.
- Salesforce Lead→Contact field mapping in Object Manager.
- Stripe dashboard setting (rare — most Stripe work is code-only).
- GTM tag, trigger, or variable saved + published.

**Code edit** = anything `git diff` shows: script tags, hidden inputs, `identify()` calls, metadata parameters in `stripe.customers.create`.

When in doubt, prefer listing something as an external checkpoint — over-tracking is cheaper than under-tracking on a resume.

## Writing rules

1. **A skill only writes to its own `## <skill-name>` section.** Never modify another skill's section. The only exception is the router, which owns `## Detected stack`.
2. **Append-only during a skill run.** Within a skill, add new `- [x]` lines as checkpoints are confirmed. Do not remove or reorder prior lines.
3. **Flip status on completion.** When every item is `[x]`, rewrite the section header from `<skill-name> — in progress` to `<skill-name> — done YYYY-MM-DD`.
4. **Do not touch completed sections on re-run.** If a skill's section is already `done` when resumed, do not touch it (the resume flow routes around it — see below).
5. **Skip already-checked items on resume.** When re-invoked on an `in progress` section, skip every `[x]` item and continue from the first `[ ]`. External checkpoints (CRM portal mutations, GA4 admin toggles, Salesforce field creation) are typically irreversible and silent on retry — do not re-do them.
6. **Dates are local absolute dates**, written as `YYYY-MM-DD`. No time components, no relative strings.

## Reading rules (router)

On resume (file exists), the router:

1. Reads `## Detected stack` and uses it in place of running Detection pass again.
2. Collects every `## <skill-name> — <status>` section.
3. Prints a summary: completed skills under "Already done", unfinished skills under "In progress" with their unchecked items.
4. Asks the user: `Continue from here? (yes / redo <skill-name> / start over)`.
   - **yes** — during Dispatch, skip every `done` skill; resume every `in progress` skill from its first unchecked item.
   - **redo <skill-name>** — delete that section from the file; treat that skill as not-yet-run.
   - **start over** — delete `.roadway/onboarding.md` entirely and restart from Detection pass.

The router never re-runs the Detection pass if `## Detected stack` is present. If the customer's stack has actually changed (new CRM, new analytics), they use `start over`.
